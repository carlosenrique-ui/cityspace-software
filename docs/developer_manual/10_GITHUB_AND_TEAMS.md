# GitHub and Microsoft Teams

---

## 1. Purpose

GitHub and Microsoft Teams constitute the collaboration layer of the Globe-CitySpace engineering environment.

Their combined objective is to provide:

- source code management;
- version control;
- documentation;
- issue tracking;
- collaboration;
- communication;
- project coordination.

Together they support the complete software engineering lifecycle.

---

## 2. Development Philosophy

The project follows a collaborative engineering model.

Every modification should be:

- documented;
- versioned;
- traceable;
- reproducible;
- reviewable.

No engineering artifact should exist only on a local workstation.

---

## 3. GitHub Repository

GitHub stores all engineering assets.

Typical repository contents include:

- source code;
- documentation;
- configuration files;
- scripts;
- engineering contracts;
- scientific metadata;
- diagrams.

GitHub becomes the project's single source of truth.

---

## 4. Version Control

All project evolution is managed through Git.

Typical operations include:

- clone;
- pull;
- commit;
- push;
- branch;
- merge;
- tag.

These operations preserve the complete engineering history.

---

## 5. Branch Strategy

Development should occur using clearly defined branches.

Typical branches include:

- main (or master);
- development;
- feature branches;
- bug-fix branches;
- release branches.

Each branch has a well-defined engineering objective.

---

## 6. Commit Philosophy

Every commit should represent a coherent engineering modification.

Good commits are:

- small;
- descriptive;
- reproducible;
- independently understandable.

Commit messages should clearly explain the engineering purpose of the change.

---

## 7. Documentation

Engineering documentation evolves together with the software.

Documentation includes:

- architecture;
- developer manual;
- operational procedures;
- engineering contracts;
- scientific workflow;
- installation guides.

Documentation receives the same version control treatment as source code.

---

## 8. Repository Organization

The repository should maintain a consistent engineering structure.

Typical top-level directories include:

- docs;
- source code;
- scripts;
- configuration;
- scientific data definitions;
- engineering contracts;
- examples.

A predictable directory structure simplifies long-term maintenance.

---

## 9. Engineering Documentation

Major engineering decisions should be documented.

Examples include:

- architectural decisions;
- coordinate systems;
- engineering contracts;
- project rotation policy;
- actuator specifications;
- scientific workflow.

Documenting decisions prevents knowledge loss over time.

---

## 10. Pull Requests

Engineering modifications should preferably be integrated through Pull Requests.

A Pull Request allows:

- code review;
- discussion;
- documentation review;
- architecture validation;
- change approval.

This process improves software quality and project traceability.

---

## 11. Code Review

Code reviews verify both software quality and engineering consistency.

Typical review topics include:

- readability;
- maintainability;
- architecture compliance;
- documentation completeness;
- scientific consistency.

Reviews reduce long-term technical debt.

---

## 12. Release Management

Stable project versions should be identified through releases.

A release normally includes:

- version identifier;
- release notes;
- supported features;
- known limitations;
- documentation updates.

This provides stable reference points throughout the project's evolution.

---

## 13. Issue Tracking

GitHub Issues record engineering tasks and project evolution.

Typical issue categories include:

- bugs;
- feature requests;
- documentation;
- scientific improvements;
- infrastructure;
- user interface.

Issues provide a structured engineering backlog.

---

## 14. Microsoft Teams

Microsoft Teams complements GitHub by supporting daily communication.

Typical uses include:

- technical meetings;
- engineering discussions;
- file sharing;
- project coordination;
- presentation of results;
- meeting recordings.

Teams manages communication while GitHub manages engineering artifacts.

---

## 15. Engineering Communication

Engineering communication should always reference version-controlled artifacts.

Typical references include:

- commit hash;
- branch name;
- issue number;
- pull request;
- release version.

This practice avoids ambiguity during project discussions.

---

## 16. Documentation Workflow

Documentation should evolve together with the software.

A recommended workflow is:

1. Implement the engineering change.
2. Update the documentation.
3. Validate the documentation.
4. Commit both code and documentation.
5. Push to the repository.

This guarantees that documentation always reflects the current implementation.

---

## 17. Engineering Traceability

Every engineering artifact should be traceable.

Examples include:

- source code;
- technical reports;
- diagrams;
- engineering contracts;
- configuration files;
- scientific documentation.

Traceability is essential for scientific reproducibility.

---

## 18. Backup Strategy

GitHub serves as the primary engineering backup.

Recommended practices include:

- frequent commits;
- regular pushes;
- tagged releases;
- multiple collaborators;
- protected main branch.

Local copies should never be considered the only backup.

---

## 19. Documentation Repository

The Developer Manual forms part of the repository itself.

Typical documentation includes:

- engineering handbook;
- architectural documentation;
- installation guide;
- operational guide;
- scientific workflow;
- roadmap.

Documentation should remain synchronized with the corresponding software version.

---

## 20. Knowledge Preservation

One objective of the repository is preserving engineering knowledge.

Knowledge includes:

- software architecture;
- scientific assumptions;
- engineering decisions;
- operational procedures;
- implementation history.

This enables future engineers to continue development without losing project context.

## 21. Engineering Collaboration

The Globe-CitySpace project is intended to support collaborative engineering.

Multiple contributors may simultaneously work on:

- software development;

- hardware integration;
- documentation;

GitHub provides the mechanisms necessary to coordinate these parallel activities.

## 22. Continuous Improvement

The repository should evolve continuously.

Typical improvements include:

Continuous improvement ensures long-term project sustainability.

## 23. Engineering Standards

These standards apply to:

## 25. Engineering Deliverables

## 26. Project Transparency

Using GitHub allows complete transparency of project evolution.

Every engineering modification can be associated with:

- commit history.

## 27. Teams Meetings

Microsoft Teams meetings should generate engineering records whenever appropriate.

Typical outputs include:

- meeting notes;

## 28. Integration Philosophy

GitHub stores the engineering knowledge.

Microsoft Teams supports engineering communication.

Together they provide a complete collaborative environment for the Globe-CitySpace project.

---

## 29. Recommended Repository Policy

The GitHub repository should contain only engineering artifacts.

Recommended contents include:

- source code;
- documentation;
- engineering contracts;
- startup scripts;
- configuration files;
- unit tests;
- integration tests.

Large scientific datasets should remain outside the repository.

---

## 30. Microsoft Teams Repository

Microsoft Teams is the recommended repository for large scientific datasets.

Typical directory structure:

```text
Scientific_Data/

├── Projects/
├── DSM/
├── DTM/
├── Orthophotos/
├── Buildings/
├── Scientific_Products/
├── Reports/
├── Metadata/
├── Demonstrations/
└── Backups/
```

This organization separates software development from scientific data management.

## 31. Engineering Backup

A complete engineering backup consists of:

GitHub

- source code;

Microsoft Teams

- scientific datasets;

---

## 32. Disaster Recovery

A new engineer should be capable of rebuilding the complete development environment using only:

- GitHub repository;
- Microsoft Teams scientific repository;
- Developer Manual.

No undocumented procedures should be required.

This requirement guarantees long-term maintainability.

---

## 33. Engineering Onboarding

The onboarding sequence for a new developer is:

1. Clone the GitHub repository.
2. Read the Developer Manual.
3. Configure the development environment.
4. Download the scientific repository from Microsoft Teams.
5. Validate the installation.
6. Execute the demonstration scripts.
7. Begin software development.

This process minimizes onboarding time.

---

## 34. Knowledge Transfer

The combination of GitHub, Microsoft Teams and the Developer Manual forms the official knowledge transfer strategy for Globe-CitySpace.

Future developers should be able to continue the project without depending on the original authors.

Engineering knowledge becomes institutional rather than individual.

---

## 35. Long-Term Sustainability

The Globe-CitySpace project has been organized to remain maintainable over many years.

Its sustainability depends on:

- clear architecture;
- complete documentation;
- version-controlled software;
- structured scientific data;
- reproducible environments;
- engineering standards.

These principles reduce dependence on individual developers.
