# Globe-CitySpace Engineering Handbook

# Chapter 12

# Roadmap

---

## 1. Purpose

This chapter presents the long-term roadmap for the Globe-CitySpace platform.

Rather than describing only software development, this roadmap integrates:

- scientific evolution;
- software engineering;
- digital twin technologies;
- tangible interfaces;
- geospatial data acquisition;
- urban simulation;
- operational deployment.

The objective is to provide a clear vision of how the platform is expected to evolve over the coming years.

---

## 2. Engineering Philosophy

The Globe-CitySpace roadmap follows several engineering principles.

The platform should remain:

- modular;
- scalable;
- reproducible;
- scientifically rigorous;
- independent of proprietary infrastructures whenever possible.

Each development phase builds upon validated engineering contracts rather than ad hoc implementations.

---

## 3. Strategic Vision

The long-term vision is to create a complete engineering platform capable of transforming any geographic location into a tangible digital twin.

Starting from only:

- latitude;
- longitude;

the platform should automatically generate:

- scientific datasets;
- digital terrain model;
- building representation;
- exploration interface;
- projection mapping;
- tangible table visualization.

This process should become increasingly automated as the platform evolves.

---

## 4. Main Development Axes

The roadmap is organized around five major axes.

### Scientific Data

Acquisition, processing and validation of geospatial datasets.

### Software Engineering

Architecture, modularization and reproducibility.

### User Experience

Exploration interfaces and interaction.

### Tangible Digital Twin

Physical table synchronization and projection mapping.

### Operational Deployment

Installation, maintenance and institutional adoption.

These axes evolve together throughout the project lifecycle.

---

## 5. Current Status

At the current stage, Globe-CitySpace already includes:

- scientific raster processing;
- Globe-CitySpace explorer;
- IPT-CitySpace projection environment;
- virtual tangible table;
- Docker execution environment;
- engineering documentation;
- GitHub repository;
- scientific repository organization.

These components establish the foundation for future developments.


## 6. Phase 1 — Scientific Foundation

The first development phase establishes the scientific basis of the platform.

Objectives include:

- scientific raster processing;
- DEM acquisition;
- DSM integration;
- orthophoto support;
- building footprint processing;
- coordinate validation;
- metadata generation.

Deliverables:

- reproducible scientific pipeline;
- validated raster products;
- engineering contracts;
- scientific documentation.

---

## 7. Phase 2 — Globe-CitySpace Explorer

The second phase focuses on user interaction.

Capabilities include:

- global navigation;
- latitude/longitude input;
- predefined locations;
- camera control;
- grid definition;
- area selection;
- rotation control;
- scientific visualization.

The Explorer becomes the primary interface for defining study areas.

---

## 8. Phase 3 — Automatic Data Generation

The next objective is to automate scientific data generation.

Starting from a selected geographic location, the platform should automatically:

- download terrain data;
- retrieve building footprints;
- compute terrain elevation;
- estimate building heights when necessary;
- generate engineering metadata;
- prepare projection products.

Automation reduces manual intervention while preserving reproducibility.

---

## 9. Phase 4 — Offline Scientific Pipeline

The offline pipeline prepares all information required for physical visualization.

Typical outputs include:

- height maps;
- projection rasters;
- engineering metadata;
- actuator plans;
- grid matrices;
- BMP images;
- CSV files.

These products are generated before real-time execution.

---

## 10. Phase 5 — IPT-CitySpace

The IPT-CitySpace interface transforms scientific products into a projection-ready environment.

Main capabilities include:

- 2D visualization;
- scientific scale display;
- projection mapping;
- grid rendering;
- north orientation indicator;
- engineering overlays.

The interface serves as the bridge between scientific processing and the tangible table.


## 11. Phase 6 — Tangible Table Integration

This phase integrates the virtual environment with the physical tangible table.

Objectives include:

- actuator synchronization;
- projection mapping;
- height reproduction;
- real-time visualization;
- engineering validation.

The virtual and physical environments should remain synchronized throughout the demonstration.

---

## 12. Phase 7 — Real-Time Operation

After the offline products are generated, the platform enters operational mode.

Capabilities include:

- real-time updates;
- actuator control;
- scenario playback;
- simulation synchronization;
- operational monitoring.

This phase supports demonstrations and decision-making activities.

---

## 13. Phase 8 — Advanced Scientific Data

Future developments include integration with additional scientific datasets.

Examples:

- LiDAR;
- high-resolution DSMs;
- municipal DTM repositories;
- cadastral information;
- transportation networks;
- environmental monitoring;
- climate indicators.

Higher-quality data directly improves the fidelity of the digital twin.

---

## 14. Phase 9 — Automation

One of the principal long-term objectives is complete automation.

Desired workflow:

```
Latitude

↓

Longitude

↓

Automatic Data Acquisition

↓

Scientific Processing

↓

Raster Generation

↓

Projection Mapping

↓

Tangible Table

↓

Operational Demonstration
```

This workflow minimizes manual configuration while preserving engineering reproducibility.

---

## 15. Phase 10 — Global Coverage

The platform should eventually operate anywhere in the world.

Depending on data availability, Globe-CitySpace should automatically combine:

- global DEMs;
- global building datasets;
- municipal datasets;
- institutional repositories;
- locally acquired drone surveys.

This multi-source strategy maximizes worldwide applicability.


## 16. Phase 11 — Urban Digital Twin

The long-term objective is the creation of complete urban digital twins.

These models should integrate:

- terrain;
- buildings;
- transportation;
- utilities;
- environmental data;
- simulation models;
- tangible visualization.

The digital twin becomes a unified engineering platform for urban analysis.

---

## 17. Phase 12 — Simulation Integration

Future versions should support integration with external simulation platforms.

Examples include:

- traffic simulation;
- pedestrian simulation;
- flood simulation;
- environmental models;
- energy systems;
- climate analysis;
- emergency response.

Simulation results can then be visualized directly on the tangible table.
