# IPT-CitySpace

---

## 1. Purpose

IPT-CitySpace is the engineering environment responsible for transforming scientific geographic information into a tangible interactive model.

It receives the outputs generated by the Offline Pipeline and converts them into a physical representation that can be visualized, manipulated and synchronized with the projection-mapped table.

The objective is to create a scientific digital twin capable of interacting with both virtual and physical environments.

---

## 2. Position within the Architecture

Within the complete Globe-CitySpace ecosystem, IPT-CitySpace is positioned after the Offline Pipeline.

The overall workflow is:

Globe-CitySpace

↓

Offline Pipeline

↓

IPT-CitySpace

↓

Virtual Table

↓

Physical Table

↓

Projection Mapping

---

## 3. Responsibilities

IPT-CitySpace is responsible for:

- loading scientific products;
- loading engineering metadata;
- reconstructing the actuator matrix;
- generating the virtual table;
- synchronizing physical actuators;
- controlling visualization;
- exporting operational information.

Unlike Globe-CitySpace, IPT-CitySpace no longer performs geographic exploration.

---

## 4. Scientific Inputs

The application receives the scientific products generated previously.

Typical inputs include:

- projection raster;
- terrain raster;
- building raster;
- engineering grid;
- metadata;
- actuator matrix;
- height matrix;
- project orientation.

These products remain immutable during operation.

---

## 5. Engineering Philosophy

IPT-CitySpace separates scientific computation from visualization.

Scientific processing has already been completed before this stage.

Therefore IPT-CitySpace performs:

- visualization;
- synchronization;
- interaction;
- presentation;
- actuator control.

No scientific calculations are modified during execution.

---

## 6. Engineering Coordinate System

The engineering coordinate system is inherited directly from Globe-CitySpace.

The stored project orientation is preserved.

Consequently:

- the engineering grid remains rotated;
- actuator positions remain fixed;
- projection remains aligned;
- metadata remain unchanged.

This guarantees complete consistency throughout the system.

---

## 7. Rotation Preservation

A fundamental design decision is that IPT-CitySpace never recomputes the project orientation.

Instead, it reads the rotation angle stored during Globe-CitySpace exploration.

Example:

Rotation = –26.02°

This angle becomes part of the operational project configuration.

---

## 8. User Interface

The IPT-CitySpace interface has been designed for engineering operation rather than GIS editing.

The main interface provides:

- virtual table visualization;
- projection visualization;
- actuator status;
- engineering controls;
- project information.

The interface remains synchronized throughout execution.

---

## 9. Virtual Table

The Virtual Table represents the digital counterpart of the physical actuator matrix.

Each virtual actuator corresponds exactly to one physical actuator.

Characteristics include:

- identical dimensions;
- identical indexing;
- identical heights;
- identical movement sequence.

The Virtual Table is the primary debugging environment.

---

## 10. Physical Table

The Physical Table consists of the real actuator matrix.

Its geometry is defined by the engineering contract.

Typical parameters include:

- number of rows;
- number of columns;
- actuator spacing;
- actuator height;
- travel limits.

The physical hardware reproduces exactly the virtual model.

---

## 11. Projection Mapping

Projection Mapping overlays scientific information directly onto the actuator surface.

Projected information may include:

- orthophotos;
- terrain shading;
- building footprints;
- thematic layers;
- simulation results;
- engineering annotations.

Projection is synchronized with actuator movement.

---

## 12. Actuator Matrix

The actuator matrix is reconstructed from the scientific grid.

Each actuator receives:

- row index;
- column index;
- terrain elevation;
- building elevation;
- total elevation.

The actuator height corresponds to the processed scientific value.

---

## 13. Height Model

The engineering height model follows the project contract.

For each actuator:

Total Height = Terrain Height + Building Height

The resulting value is converted into actuator displacement according to the selected physical scale.

No additional interpretation is performed during operation.

---

## 14. Engineering Synchronization

Synchronization occurs simultaneously between:

- scientific data;
- virtual table;
- physical table;
- projection mapping.

Every component references the same engineering dataset.

This guarantees that all representations remain spatially consistent.

---

## 21. Project Loading

When IPT-CitySpace starts, it loads the complete engineering project generated by the Offline Pipeline.

The loading sequence is:

1. Read project metadata.
2. Load scientific rasters.
3. Load engineering grid.
4. Load actuator matrix.
5. Initialize the Virtual Table.
6. Initialize Projection Mapping.
7. Establish synchronization.

After this sequence, the project is ready for operation.

---

## 22. Virtual and Physical Synchronization

Synchronization is event-driven.

Whenever the actuator matrix changes:

- the Virtual Table is updated;
- the Physical Table receives the same commands;
- Projection Mapping refreshes automatically.

This architecture guarantees that all representations remain identical.

---

## 23. Engineering Contracts

IPT-CitySpace follows predefined engineering contracts that describe the physical table.

Typical contracts define:

- actuator count;
- actuator positions;
- actuator numbering;
- zigzag traversal order;
- maximum stroke;
- physical dimensions;
- table origin.

These contracts are independent from the scientific data.

---

## 24. Zigzag Traversal

The actuator activation order follows the engineering zigzag contract.

The traversal order is deterministic and reproducible.

Its objectives include:

- minimizing actuator travel;
- simplifying wiring;
- reducing controller complexity;
- ensuring repeatable demonstrations.

The traversal order is stored together with the project configuration.

---

## 25. Projection Alignment

Projection Mapping is aligned using the same engineering reference adopted during Globe-CitySpace exploration.

The projected image, engineering grid and actuator matrix share:

- identical origin;
- identical scale;
- identical rotation.

This guarantees precise visual correspondence between projected information and physical actuator positions.

---

## 26. Operational Safety

Operational limits are enforced before actuator commands are executed.

Typical validations include:

- maximum actuator height;
- minimum actuator height;
- matrix dimensions;
- valid engineering metadata;
- synchronization status.

Invalid configurations are rejected before movement begins.

---

## 27. Visualization Layers

Different visualization layers may be enabled independently.

Examples include:

- orthophoto;

- building model;
- engineering grid;
- actuator identifiers;
- scientific annotations.

Layer visibility does not affect the underlying engineering model.

IPT-CitySpace prioritizes smooth and deterministic execution.

Performance objectives include:

The Demonstration Mode allows the complete engineering workflow to be presented without requiring modifications to the scientific products.

## 31. Integration with the Explorer

IPT-CitySpace does not redefine project parameters.

Instead, it consumes the engineering products produced by Globe-CitySpace and the Offline Pipeline.

The transferred parameters include:

- project centroid;
- project boundary;
- engineering grid;
- project rotation angle;
- scientific metadata.

This guarantees that every engineering decision made during exploration is preserved.
