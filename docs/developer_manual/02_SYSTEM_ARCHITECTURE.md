# 1. Purpose

This chapter presents the complete software architecture of the Globe-CitySpace platform.

The objective is to describe how the different software components interact to transform geographic information into scientific products and tangible representations.

The architecture was designed following software engineering principles, emphasizing modularity, reproducibility, maintainability and extensibility.

---

# 2. Architectural Vision

Globe-CitySpace is organized as an engineering platform composed of independent subsystems.

Each subsystem performs a specific responsibility while communicating through clearly defined interfaces.

This architecture allows new capabilities to be incorporated without affecting the remaining components.

The platform is divided into two major domains:

- Offline Scientific Domain
- Online Runtime Domain

These domains communicate through standardized scientific products generated by the offline processing pipeline.

# 4. Architectural Principles

The architecture follows the principles below.

## 4.1 Modularity

Each subsystem performs a single engineering responsibility.

Subsystems communicate through documented interfaces.

---

## 4.2 Scientific Reproducibility

The scientific pipeline must always generate identical outputs from identical inputs.

Scientific reproducibility is considered a mandatory requirement.

---

## 4.3 Separation of Concerns

The software separates:

- visualization;
- scientific processing;
- runtime execution;
- projection mapping;
- documentation.

This separation simplifies maintenance and future development.

---

## 4.4 Extensibility

New geographic datasets, cities, scientific algorithms and visualization modules can be incorporated without redesigning the platform.

---

## 4.5 Maintainability

The software architecture prioritizes readability, documentation and long-term maintenance.

Every major subsystem is documented independently within this handbook.

# 9. Globe-CitySpace Subsystem

The Globe-CitySpace subsystem represents the entry point of the platform.

It provides the user with an interactive global environment where geographic information can be explored before scientific processing begins.

Its responsibilities include:

- worldwide navigation;
- city selection;
- latitude and longitude search;
- camera control;
- zoom management;
- cartographic visualization;
- Area of Interest (AOI) definition.

The Globe-CitySpace subsystem is independent of the scientific pipeline.

Its only responsibility is defining **where** scientific processing will occur.

---

# 10. Area of Interest (AOI)

The Area of Interest represents the geographic region selected by the user.

The AOI is the contract between the visualization subsystem and the scientific pipeline.

Once defined, the AOI becomes immutable during scientific processing.

Typical AOI parameters include:

- center latitude;
- center longitude;
- cartographic scale;
- grid dimensions;
- projection parameters.

The AOI becomes the reference for every subsequent engineering process.

---

# 11. Offline Scientific Pipeline

The Offline Scientific Pipeline is the scientific core of Globe-CitySpace.

Its responsibility is transforming raw geographic information into engineering products.

Unlike the visualization subsystem, the pipeline performs deterministic scientific calculations.

Its execution is independent of the user interface.

The pipeline can therefore be executed automatically or integrated into batch workflows.

---

# 14. Scientific Outputs

The scientific pipeline generates several engineering products.

Typical outputs include:

- Cartographic Grid;
- Height Maps;
- Gray-scale Maps;
- BMP Files;
- CSV Files;
- Metadata Files;
- Scientific Reports;
- Projection Mapping Products;
- Actuator Plans.

# 15. Scientific Product Contracts

Every scientific product follows predefined contracts.

Typical contracts define:

- dimensions;
- coordinate systems;
- raster orientation;
- numeric precision;
- metadata;
- file naming conventions.

Using contracts guarantees compatibility between independent software modules.

---

# 16. Reproducibility

Scientific reproducibility is considered mandatory.

Processing the same geographic region with identical parameters must always generate identical engineering products.

This principle guarantees:

- repeatability;
- validation;
- scientific comparison;
- long-term maintenance.

---

# 17. Engineering Validation

Before products are transferred to the runtime environment they undergo validation procedures.

Typical validation includes:

- raster dimensions;
- coordinate consistency;
- grid integrity;
- metadata verification;
- product completeness.

Only validated products are accepted by the Online Runtime.

---

**BLOCO 2 DE 7**

# 18. Online Runtime

The Online Runtime is responsible for transforming scientific products into interactive visualizations and tangible representations.

Unlike the Offline Pipeline, which performs scientific computation, the Online Runtime focuses on user interaction and real-time operation.

Its responsibilities include:

- loading scientific products;
- visualization;
- projection mapping;
- synchronization;
- user interaction;
- actuator control.

---

# 19. IPT-CitySpace

IPT-CitySpace represents the operational environment of the platform.

It receives validated scientific products generated by the Offline Pipeline and transforms them into interactive visualizations.

Its main responsibilities include:

- loading scientific datasets;
- rendering projection maps;
- displaying scientific layers;
- integrating virtual and physical tables;
- supporting engineering demonstrations.

The runtime never modifies scientific products.

It only consumes validated outputs.

---

# 20. Projection Mapping

Projection Mapping transforms scientific raster products into projected visual information.

Projection Mapping uses:

- cartographic grids;
- grayscale height maps;
- scientific masks;
- projection calibration;
- visualization layers.

The projected representation preserves the spatial correspondence established during the Offline Pipeline.

---

# 21. Virtual Table

The Virtual Table simulates the behavior of the physical tangible interface.

Its objectives include:

- validating engineering workflows;
- testing actuator sequences;
- verifying projection alignment;
- supporting demonstrations.

Because the Virtual Table reproduces the Physical Table behavior, software validation can be performed before hardware execution.

---

# 22. Physical Table

The Physical Table is the tangible interface of Globe-CitySpace.

It converts digital scientific information into physical representations.

Typical components include:

- actuators;
- pins;
- projection system;
- synchronization software;
- runtime controller.

The Physical Table receives commands exclusively from validated scientific products.

---

# 23. Synchronization

Synchronization guarantees consistency between:

- Globe-CitySpace;
- Offline Scientific Pipeline;
- IPT-CitySpace;
- Projection Mapping;
- Virtual Table;
- Physical Table.

Every subsystem shares the same scientific reference generated by the Offline Pipeline.

# 25. Runtime Responsibilities

The Online Runtime is responsible for:

- loading products;
- rendering layers;
- displaying scientific information;
- executing simulations;
- synchronizing interfaces;
- communicating with hardware.

Scientific computation is intentionally excluded from the runtime.

---

# 26. Runtime Independence

The Online Runtime depends only on scientific products.

It does not require access to:

- original DSM;
- original DTM;
- raw rasters;
- preprocessing algorithms.

This separation reduces complexity and improves maintainability.

---

- Virtual Table.

This integrated workflow allows users to navigate, process, visualize and interact with geographic information in a unified engineering environment.

- Modular Design
- Separation of Concerns

- Configuration over Hard Coding
- Contract-Based Communication

These principles simplify future maintenance and extension of the platform.

# 45. Engineering Contracts

Communication between independent modules is based on engineering contracts.

- file formats;
- coordinate systems;
- raster dimensions;
- metadata;

- directory organization.

# 46. Runtime Independence

One important architectural objective is ensuring that runtime visualization never depends directly on scientific processing.

Instead, runtime components consume only validated scientific products.

This separation enables:

- independent testing;
- simplified deployment;
- faster demonstrations;
- reproducible results.

# 47. Maintainability Strategy

Long-term maintainability is considered a primary architectural requirement.

The project adopts:

- source code documentation;
- engineering handbook;
- standardized directory organization;
- version control;
- modular components;
- documented workflows.

Future contributors should be able to understand the platform without relying on its original developers.

---

# 48. Extensibility Strategy

New functionality should be incorporated through new modules instead of modifying existing stable components whenever possible.

Examples include:

- additional cities;
- new scientific datasets;
- alternative visualization engines;
- new actuator technologies;
- additional Digital Twin capabilities.

This strategy minimizes regression risks.
